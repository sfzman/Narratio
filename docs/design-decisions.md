# docs/design-decisions.md — 设计决策记录

> 格式：每条决策标注 [DECIDED] 或 [OPEN]。
> Agent 遇到不确定的设计问题时，在此记录并标注 [OPEN]，不得自行假设。

---

## [DECIDED] 调度模型：Job 下挂 Task DAG

**决策**：一个用户请求对应一个 `job`，`job` 内由多个 `task` 组成 DAG；调度和并发控制以 task 为单位进行。

**理由**：
- 后续工作流不会长期停留在固定四阶段
- 大纲、人物表、分镜等中间节点需要可复用的表达方式
- 无依赖 task 可以并行，提高整体吞吐

---

## [DECIDED] 任务队列：内存队列 vs 持久化队列

**决策**：MVP 阶段使用内存 channel 队列，不引入 Redis/MQ。

**理由**：
- 初期用户量小，重启丢失任务可接受
- 降低基础设施复杂度
- 后期可替换为 Asynq（基于 Redis）而不改变 scheduler / executor 语义

**影响**：服务重启时进行中的任务状态变为 failed，前端需提示用户重新提交。

---

## [DECIDED] 状态存储：统一 SQL Store，生产 MySQL，本地 SQLite

**决策**：MVP 阶段不做内存 store，直接使用 SQL store。

- 本地开发和测试：SQLite
- 线上部署：MySQL

**理由**：
- 调度状态、task 依赖和运行结果本身就是持久化数据，不值得先做一套临时内存模型再迁移
- 你明确计划上线到云环境，尽早按数据库约束设计更稳妥
- SQLite 足够支撑本地开发与集成测试，MySQL 则覆盖线上部署场景

**接口设计**：
- `store.JobStore` 和 `store.TaskStore` 仍定义为接口
- 提供统一 SQL 风格实现，尽量避免方言绑定在业务层扩散
- schema 设计优先兼容 SQLite 和 MySQL 的公共子集

**约束**：
- 不在业务 SQL 中依赖某一数据库专属特性，除非在文档中显式记录
- migration 必须纳入仓库管理，不能靠手工建表

## [DECIDED] 并发控制：按资源池限流，不按 TaskType 限流

**决策**：调度器按 `ResourceKey` 控制并发，不直接按 `TaskType` 控制。

**示例**：
- `outline` 和 `character_sheet` 共享 `llm_text`
- `image` 使用 `image_gen`
- `video` 使用 `video_render`

**理由**：
- 不同 task type 可能实际占用同一套模型或算力
- 资源池更贴近真实系统瓶颈
- 方便后续动态调整限额

---

## [DECIDED] 视频格式

**决策**：统一输出 MP4 (H.264 + AAC)，帧率 24fps；画幅先支持 `16:9` 与 `9:16` 两档，默认 `9:16`。

**理由**：兼容移动端短视频竖屏场景，同时保留横屏导出能力；编码格式仍保持 Web / 移动端兼容性。

---

## [DECIDED] 图像生成失败不阻断流程

**决策**：单个 segment 图像生成失败时，使用纯色背景降级，不中止整个 job。

**理由**：图像生成是辅助元素，语音是核心。宁可有个简陋视频，也不要让用户完全失败。

**降级颜色**：`#1a1a2e`（深蓝色，视觉中性）

---

## [DECIDED] 前端轮询 vs WebSocket

**决策**：使用轮询，不用 WebSocket。

**理由**：视频生成耗时 1~3 分钟，轮询间隔 3~10s 完全够用，WebSocket 增加服务端复杂度收益有限。

---

## [OPEN] 多用户隔离

**问题**：当前设计没有用户系统，任意人都能查询/下载所有 job。

**待决策**：
- 方案 A：不做隔离，MVP 阶段内部使用
- 方案 B：job_id 使用 UUID v4（随机，难猜），当作 token 使用
- 方案 C：引入简单的 API Key 认证

**建议**：先用方案 B，上线前再评估是否需要方案 C。

---

## [DECIDED] 顶层 Job 状态与 Task 状态分离

**决策**：顶层 `job.status` 只表示生命周期，执行细节放在 `task.status` 聚合结果中。

**生命周期状态**：`queued`、`running`、`cancelling`、`completed`、`failed`、`cancelled`

**理由**：
- 避免 `tts_done` 这类值同时承担“任务是否完成”和“当前处于哪个阶段”两种语义
- 前端轮询逻辑更简单
- 取消逻辑可以自然引入 `cancelling` / `cancelled`

---

## [DECIDED] 字幕先做 segment 级，不做逐词对齐

**决策**：MVP 只输出 segment 级字幕时间轴，每段一条字幕。

**理由**：
- 当前 TTS 契约只有 segment 级时长，没有词级时间戳
- 逐词高亮需要额外对齐器或支持时间戳的 TTS，超出首版范围
- segment 级字幕已足以验证 Web App 端到端体验

---

## [OPEN] Workspace 清理策略

**问题**：生成的音频、图片、视频文件如何清理？

**待决策**：
- 任务完成后 24h 删除
- 用户主动下载后立即删除
- 定时任务每天清理 7 天前的文件

## [OPEN] 朗诵脚本的停顿符号规范

**问题**：使用 `|` 和 `||` 表示短/长停顿，但 TTS 服务是否能识别这些符号？

**待确认**：调通自部署 TTS API 后，测试符号处理方式。若 TTS 不支持，需要在 tts 模块中将停顿符转换为 SSML 或 TTS 支持的格式。
